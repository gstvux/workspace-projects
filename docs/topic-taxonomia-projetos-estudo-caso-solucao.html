Estudo de Caso & Resolução

Evolução da Estrutura de Pastas e Nomenclatura

Como transcender a divisão técnica binária para criar um sistema de governança escalável, padronizado por prefixos orientados a propósito.

A

A Dúvida Inicial: O Caso Limite (Edge Case)

Ao planejar a transição de pastas genéricas (como frontend, backend, automations e sandbox) para um modelo de pasta única com prefixo (prefixo-nome_projeto), surge uma fragilidade conceitual importante ao classificar certos projetos modernos.

O Cenário Analisado

Imagine um projeto que entrega um site estático construído em Next.js/React, utilizando TinaCMS rodando localmente (onde a publicação ocorre via commit direto das alterações estáticas). Além disso, o repositório possui uma ferramenta acoplada para gerenciar as versões do currículo e manter o conteúdo atualizado editorialmente.

As perguntas fundamentais de Governança:

  • Este projeto é um frontend ou um híbrido?
  • Como classificar essa pasta sem tornar as regras de nomenclatura frágeis?
  • Qual abordagem de governança nos dá flexibilidade técnica sem perder a capacidade de automatizar scripts futuros?
B

Raciocínio & Análise Arquitetural

A fragilidade do modelo antigo reside em tentar catalogar aplicações pelo seu "meio de entrega técnico" (onde o código roda) em vez do seu "propósito de domínio". Abaixo estão os pilares conceituais para desatar esse nó:

O Conflito de Meta-Frameworks

Plataformas modernas como Next.js não são apenas frontends. Elas gerenciam builds locais, executam funções de servidor, geram arquivos estáticos e integram CMSs. Tratá-las como mero "frontend" ignora sua natureza full-stack e integrada.

A Regra de Ouro da Coesão

A ferramenta de currículo e o TinaCMS mudam em sincronia com o site? Sim. Eles existem isoladamente para outras aplicações? Não. Portanto, pela regra de alta coesão, eles devem morar no mesmo repositório e pasta, tratados sob um nome unificado.

Estabilidade para Automações

Nossos scripts precisam ler o prefixo da pasta e saber imediatamente como executar tarefas. Um prefixo como app- avisa à automação: "este é um web app full-stack/integrado, faça build local e publique os artefatos estáticos".

Governança Escalável

Ao definir diretrizes técnicas claras de enquadramento, eliminamos debates subjetivos. Criamos um processo reprodutível de classificação que apoia o crescimento sem reestruturar diretórios repetidamente.

C

Evolução da Taxonomia (Nomenclatura)

Para solucionar a fragilidade do modelo inicial, expandimos e redefinimos os prefixos. A taxonomia evolui de "camadas técnicas" para "funções de domínio e entrega".

Taxonomia Proposta de Prefixos Padrão: prefixo-$project_name
app-
Aplicação Full-Stack
Uso para web apps integrados, meta-frameworks (Next.js, Remix) ou monólitos híbridos. Ex: app-gstvux.com.br (Nosso Edge Case!)
front-
Interface Pura
Aplicações estáticas sem camada de backend integrado (ex: React SPA puro que consome APIs externas). Ex: front-dashboard
api-
Serviço/API
Microsserviços de backend ou APIs isoladas (Node, Python, Go) focados puramente em processamento e dados. Ex: api-auth-service
tool-
Ferramenta/Utilitário
CLI tools, geradores, scripts de utilidade de desenvolvedores que não rodam em produção de forma contínua. Ex: tool-cv-formatter
auto-
Automações/Scripts
Cronjobs, automações de deploy, scripts autônomos de infraestrutura ou fluxos de CI/CD complexos. Ex: auto-deploy-runner
play-
Playground/Sandbox
Ambientes experimentais, laboratórios ou códigos temporários para testes livres sem obrigação de deploy rígido. Ex: play-angular-18-test
