Governança de Projetos & Padrão de Prefixos

Diretrizes técnicas e regras formais para a classificação, criação de pastas e automação do ciclo de desenvolvimento de projetos baseados na taxonomia evoluída.

1

Padrão Formal de Nomenclatura

Todo projeto adicionado ao ecossistema de workspaces deve, obrigatoriamente, seguir a nomenclatura padronizada. Esta regra garante que scripts de Integração Contínua (CI), deploys automatizados e rotinas locais localizem e gerenciem os repositórios sem a necessidade de configurações manuais redundantes.

A Regra Geral

<prefixo>-<project_name>

<prefixo>: Categoria do projeto (determina o pipeline de build/deploy).

<project_name>: Código único e imutável do projeto, em formato kebab-case lowercase.

Detalhamento dos Critérios de Uso

Use a tabela abaixo para enquadrar novos projetos conforme suas características operacionais:

Prefixo Tipo de Destino Pipeline Padrão de Deploy Exemplo Prático
app- Full-Stack / Híbridos Build local de estáticos + funções Serverless / Docker Multi-stage app-gstvux.com.br
front- SPA / Estático Puro Geração de SPA estática e publicação direta em CDN (S3, Vercel) front-curriculum-viewer
api- APIs e Microsserviços Build Docker image + deploy em Cluster de Container (K8s, ECS, VPS) api-resume-database
tool- Ferramentas Internas Apenas empacotamento local, publicação de pacote NPM/PyPI privado tool-cv-formatter
auto- Scripts de Automação Deploy em Runners locais de Cron / Workflows de CI/CD auto-sync-linkedin
play- Experimentos / Sandbox Sem pipeline de deploy configurado. Exclusivo para código local. play-nextjs-experimental
2

Árvore de Decisão para Escolha de Prefixos

Sempre que for criar um novo projeto, siga este fluxo lógico simples para identificar qual o enquadramento exato:

01 O projeto possui fins de deploy em produção?
NÃO → Use play- SIM → Avançar para Passo 2
02 O código é puramente utilitário ou automação?
Utilitário local → tool- Automação de Infra → auto- Aplicação de Negócio → Avançar para Passo 3
03 Possui código cliente (Front) E servidor (Back) acoplados?
Apenas Backend → api- Apenas Frontend → front- AMBOS (Híbrido) → Use app-
3

Implementação Prática: Script de Deploy Dinâmico

A grande vantagem de padronizar a governança é a possibilidade de automatizar processos de infraestrutura. Abaixo está um exemplo prático de um script Bash que escaneia a taxonomia do repositório no pipeline de CI/CD para disparar as tarefas apropriadas dinamicamente:

deploy-pipeline.sh 
#!/usr/bin/env bash
# Script de deploy inteligente baseado na nomenclatura de pastas

# Extrai o nome da pasta do diretório atual
FOLDER_NAME=$(basename "$PWD")

# Divide o nome da pasta com hífen como delimitador
PREFIX="${FOLDER_NAME%%-*}"
PROJECT_NAME="${FOLDER_NAME#*-}"

echo "=========================================="
echo "Iniciando Processo de Deploy Automatizado"
echo "Projeto: $PROJECT_NAME"
echo "Taxonomia Detectada: $PREFIX"
echo "=========================================="

case "$PREFIX" in
  "app")
    echo "[BUILD/DEPLOY] Processando meta-framework full-stack..."
    echo "[1/3] Executando build de páginas estáticas Next.js..."
    npm run build
    echo "[2/3] Executando rotinas locais do TinaCMS..."
    npx tinacms build
    echo "[3/3] Sincronizando e publicando arquivos..."
    # Exemplo: aws s3 sync out/ s3://$PROJECT_NAME --delete
    echo "[OK] Deploy do app concluído com sucesso!"
    ;;
    
  "front")
    echo "[BUILD/DEPLOY] Compilando interface de usuário pura..."
    npm run build:prod
    # Exemplo: deploy CDN
    echo "[OK] Frontend publicado."
    ;;
    
  "api")
    echo "[BUILD/DEPLOY] Empacotando microsserviço de API..."
    docker build -t "local/api-$PROJECT_NAME:latest" .
    # Exemplo: docker push e atualização de cluster
    echo "[OK] API/Service atualizado."
    ;;
    
  "tool")
    echo "[INFO] Ferramenta detectada. Publicando pacote local..."
    npm publish --access restricted
    ;;
    
  "auto")
    echo "[INFO] Atualizando scripts de agendamento e cronjobs..."
    # Exemplo: copiar script para pasta de cron do runner
    ;;
    
  "play")
    echo "[AVISO] Pasta classificada como Playground/Sandbox."
    echo "Deploys de produção são terminantemente ignorados."
    exit 0
    ;;
    
  *)
    echo "[ERRO CATASTRÓFICO] Pasta '$FOLDER_NAME' fora do padrão de taxonomia."
    echo "Adicione um prefixo válido (app-, front-, api-, tool-, auto-, play-)."
    exit 1
    ;;
esac

Benefícios imediatos da solução:

  • Automação Inteligente: O CI do GitHub Actions ou GitLab CI pode ler o nome do repositório e rodar o job correto automaticamente usando um único arquivo centralizado.
  • Zero Ambiguidade: Desenvolvedores novatos sabem exatamente como e onde criar seus códigos baseando-se apenas nos nomes de pastas.
  • Simplicidade Local: A pasta do desenvolvedor reflete a taxonomia sem a complexidade de criar monorepos complexos se o momento não exigir.
