Arquitetura de CLI Portátil e Reutilizável de Projeto: Caso de Estudo `gstv`

Estudo conceitual da evolução técnica e tomada de decisão para a criação do hub de automação de projetos `gstv` configurável via YAML.

1. Síntese da Colaboração

A necessidade original partiu de um problema cotidiano do desenvolvedor: possuir scripts utilitários isolados em uma pasta scripts/ e desejar dispará-los confortavelmente da raiz do projeto ou "plugá-los" no PATH do sistema de forma instantânea.

Durante o diálogo, exploramos a tensão clássica entre a poluição do sistema (adicionar múltiplos caminhos locais de desenvolvimento ao PATH) e a facilidade de uso (chamar comandos diretos sem prefixos longos). Optamos pelo padrão de **CLI centralizado e auto-documentado**, expandindo a ideia conceitual para criar um **template universal de automação** que se adapta dinamicamente a diferentes projetos por meio de metadados externos.

2. Ferramenta & Arquitetura Proposta

A arquitetura implementada baseia-se em três pilares fundamentais, todos projetados para operar com zero dependências externas:

1. project.yaml (Config)

O arquivo declarativo de configuração na raiz de cada projeto que define a chave cli_command. Ele isola a identidade do projeto do código executor.

2. O Dispatcher local

O roteador principal (o script executável gstv) encarregado de parsear nativamente o YAML em bash e encaminhar chamadas e parâmetros via exec "$@".

3. Relink & GC

Mecanismo integrado que limpa links simbólicos órfãos no diretório do usuário (~/.local/bin) antes de registrar a nova identidade do CLI.

Abaixo, comentamos o cerne lógico do sistema de resolução de caminhos e parsing robusto contidos no roteador: 

# Resolução dinâmica e robusta a links simbólicos (O "Symlink Trap" resolvido)
REAL_PATH="$(readlink -f "${BASH_SOURCE[0]}")"
PROJECT_ROOT="$(cd "$(dirname "$REAL_PATH")" && pwd)"
SCRIPTS_DIR="$PROJECT_ROOT/scripts"

# Leitura nativa rápida de arquivos YAML (0 dependências de npm ou pip)
CLI_NAME=$(grep -E "^cli_command:" "$PROJECT_ROOT/project.yaml" | head -n 1 | sed -E 's/^cli_command:[[:space:]]*["'\'' ]?([^"'\''[:space:]]+)["'\'' ]?.*$/\1/')
[ -z "$CLI_NAME" ] && CLI_NAME="gstv"

Registro Temporal de Decisões

18/Mai - 10:14:32

Demanda & Planejamento Inicial

Gatilho: Desejo de rodar scripts da pasta /scripts a partir da raiz do projeto e plugá-los globalmente no PATH. Solicitado planejamento prévio.
Solução: Análise do projeto. Mapeadas 4 abordagens (PATH direto, CLI p, Symlinks em ~/.local/bin e shell aliases). Identificados edge cases: "Symlink Trap" (BASH_SOURCE), line-endings Windows CRLF e colisões globais.

18/Mai - 10:18:28

Definição da Abordagem de CLI Centralizado

Gatilho: Preferência pela Opção 2 (CLI Dispatcher Centralizado) e pedido de detalhamento técnico (esquema) antes de codificar.
Solução: Estruturado o design do roteador p na raiz. Planejado o cabeçalho de auto-documentação para scripts, forwarding de argumentos (exec "$@") e mapeamento robusto.

18/Mai - 10:29:46

Configuração Dinâmica por YAML

Gatilho: Ideia de portar o CLI para outros projetos mudando a palavra de invocação global de forma ágil através de um arquivo de configuração (ex: de gstv para vela).
Solução: Integração de um arquivo project.yaml contendo cli_command. Arquitetado um parser nativo YAML puro em Bash e criados comandos internos de linkagem automática.

18/Mai - 10:45:10

Identidade Padrão: gstv

Gatilho: Correção no design: rejeição do termo genérico p como nome padrão. A identidade do roteador local e o fallback global devem ser exclusivamente gstv.
Solução: Reestruturação total do plano nos artefatos. Ajuste de nomes de variáveis, fallbacks internos e caminhos de arquivos para usar a identidade consolidada gstv.

18/Mai - 11:34:37

Comando relink e Garbage Collection

Gatilho: Caso de borda avançado na mudança dinâmica de aliases: se o CLI global ativo for vela e for alterado para teia, o comando global executado será vela relink. Isso precisa limpar o link simbólico antigo vela de forma limpa.
Solução: Renomeação do comando para relink. Desenvolvimento de um **Garbage Collector de Symlinks** no Bash que varre ~/.local/bin procurando links do mesmo projeto para apagá-los de forma dinâmica antes de registrar o novo link.

18/Mai - 11:49:01

Consolidação e Documentação

Gatilho: Aprovação para a execução prática combinada com a criação das documentações HTML de Tópico (topic-) e Referência Técnica (docs-).
Solução: Criação física dos arquivos executáveis, configuração do YAML, linkagem global, elaboração das documentações em HTML com visual leve e alta densidade técnica compacta, e registro do tópico no catálogo central.
